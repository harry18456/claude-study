## Context

本次變更的目標是建立一套完整的 Claude 功能介紹文件，供工程師同事在週會後作為參考資料。文件涵蓋 Claude 的三大使用層面：一般使用者介面（Claude.ai）、工程師開發工具（Claude Code CLI）、以及程式串接（API）。此外，還涵蓋 MCP 整合協議與 Skill 系統等進階功能。

文件將以 Markdown 格式直接存放於專案根目錄，無需建置流程或部署。

## Goals / Non-Goals

**Goals:**

- 建立 7 份完整的 Claude 功能參考文件
- 每份文件針對工程師受眾撰寫，強調實際使用情境
- 文件之間有清楚的層次關係（入門 → 進階）
- 可作為週會簡報的內容來源

**Non-Goals:**

- 不建立互動式或網頁版文件
- 不翻譯成英文（文件以繁體中文撰寫）
- 不包含實際的程式碼範例測試
- 不建立週會簡報（簡報由文件提煉，屬後續工作）

## Decisions

### 文件結構採用「功能導向」而非「使用者旅程導向」

**決策**：每份 MD 對應一個功能層面，而非依照「初學者 → 進階」的旅程排列。

**理由**：工程師受眾通常不會線性閱讀，而是依需求直接跳到對應章節。功能導向結構更易於查閱。

**替代方案**：旅程導向（BEGINNER.md → INTERMEDIATE.md → ADVANCED.md）→ 不採用，因為不同工程師的起點不同。

### CODE.md 與 API.md 分開

**決策**：Claude Code CLI 和 Claude API 分為兩份文件。

**理由**：兩者的使用對象不同。Claude Code 是「工程師作為使用者」的工具；Claude API 是「工程師作為開發者」的工具。混在一起會造成混淆。

### MCP 獨立成 MCP.md

**決策**：MCP 不歸入 CODE.md 或 API.md，獨立一份文件。

**理由**：MCP 是跨層面的協議，同時適用於 Claude.ai、Claude Code、和自建應用。放在任一個功能文件中都不完整。

## Risks / Trade-offs

- **文件過時風險** → Claude 的功能更新頻繁，文件需定期維護。緩解：文件中標註「知識截止日期」。
- **深度 vs 廣度取捨** → 7 份文件若每份都求深入，工作量龐大。緩解：每份文件以「核心功能 + 實用範例」為主，避免過度詳細的技術規格。
- **受眾假設** → 假設讀者已熟悉 AI 工具基本概念，若有同事完全不了解 LLM，GENERAL.md 需要額外補充基礎知識。
